# Daily Research: M★ — Every Task Deserves Its Own Memory Harness

## 研究動機

Path A discovery-driven：arxiv 搜 "agent memory" 前 10 篇，2604.11811 是 4/10 發的新 paper，wiki 還沒收。標題 "Every Task Deserves Its Own Memory Harness" 直接延伸 [[fluxmem]] 的 adaptive structure 思路，但更激進 — 不是在幾個固定結構中選，是讓系統自己進化。

## 發現

**核心：memory 架構不該由人類設計，而是由進化搜尋自動發現。**

- M★ 把 memory system 建模為 **executable Python program**（Schema + Logic + Instructions），用 population-based evolutionary search 自動發現 task-optimal 架構
- **最關鍵的結果**：不同 domain 進化出完全不同的 memory 架構。ALFWorld（embodied planning）進化出 97 行的確定性 action cache，完全不用 vector search；LoCoMo（對話）進化出 290 行的 hybrid SQLite + ChromaDB。Cross-task transfer 幾乎都 fail
- 7/8 benchmark configurations 最佳：LoCoMo F1 +23% over Mem0、PRBench Finance +30% over GEPA
- Ablation：移除 code evolution → F1 下降 44%，移除 instruction optimization → 下降 23%。結構（code）是主要驅動力，prompt 是 amplifier
- 穩定性好：5 random seeds，CoV < 9%，14/15 seeds 贏 strongest baseline

## 與已有知識的連結

- **[[fluxmem]]**：FLUXMEM 是 runtime routing（三選一），M★ 是 compile-time optimization（從零進化）。FLUXMEM 的 insight 是「不同 conversation 需要不同結構」，M★ 把這推到「不同 domain 需要完全不同的 memory program」
- **[[meta-harness]]**：Stanford 在 harness 層做 automated optimization（6x 差異），M★ 在 memory 層做。兩者都用 population-based search，都發現自動搜尋 > 人工設計
- **[[self-improving-agent]] / [[asg-si]]**：M★ 把 memory system 當成 verifiable artifact（Python program），是 skill-based self-improvement 的又一個實例
- **[[memwright]]**：今天剛收的 Memwright 主張 zero LLM in retrieval。有趣的是 M★ 在 ALFWorld 上自己也進化出了 zero-vector-search 的架構，殊途同歸

## Open Questions 推進

直接推進 #3（Memory Architecture Selection）：M★ 證明 architecture selection 可以被自動化，且自動搜尋的結果 consistently 贏人類設計。但 M★ 的進化成本高（offline evolution 需多輪 LLM calls），production 可行性取決於 task 是否穩定到值得 offline optimization。

也推進 #8（Memory vs Context Window）：M★ 暗示「one size fits all」的 memory + context 方案注定是 compromise — 真正的最佳解是 per-task 特化。
